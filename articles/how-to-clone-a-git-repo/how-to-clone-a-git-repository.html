Before we start

If you're new to Git, and have never used a terminal, you might have trouble understanding what are we doing. I will try my best to explain every step, but if you do not have any expirience with CLI based applications, this article might not be for you.

In this article you will learn...

  • How to install Git
  • How to clone a repository from GitHub into your computer
  • Some basics about Git

Installing Git

Before we install Git, we should check if it is not already installed. We can do that by opening up a terminal (or Powershell/CMD on Windows) and executing this command:

$git --version

If Git is not installed yet, you will see this error on Windows...

1'git' is not recognized as an internal or external command,
2operable program or batch file.

...or this error on Linux (the error message might look different based on your distribution of Linux).

1-bash: git: command not found

If Git is already installed, you will see a message like this appear in your terminal:

Windows:git version 2.45.2.windows.1
Linux (Debian):git version 2.39.5

If Git is already installed, then you can skip this step. If Git is not installed yet, we will need to install it first.

If you are on Windows, you will have download the executable (.exe) from Git's website and then install it. You can download the .exe file here.

Installing Git on Linux is much easier thanks to package managers. If you are using Linux, please enter one of these commands into your terminal, depending on your package manager:

APT (Debian):sudo apt install git-all
DNF (Fedora):sudo dnf install git-all
Pacman (Arch):pacman -S git

After installing Git, try checking Git's version using the command bellow to verify that Git has been successfully installed. Please note most Linux distributions come with Git already preinstalled! If you saw an error in your terminal, please check you typed the command correct first!

$git --version

What is a repository

A repository is like a special storage space (well, just a folder) where you keep all the important parts of a project together. Think of it as a dedicated folder on your computer or online where you store all the files, documents, and information related to a particular project. This makes it easy to organize, manage, and share your work with others.

Now imagine that you have created a project, and you want to play with it a little. You know, make a few changes here and there, and suddenly, boom! You broke the project and don't have any idea how to return it to the state it was in before.

That's where Git comes in. Once you finish one part of a project, you can save it like a checkpoint in a video game and continue coding. If you destroy something or want to return the project to the state it was in a few days ago, you can use Git for that.

So how to use Git? Quick Note: You will learn Git faster if you try using it yourself, so I recommend you to follow along. First of all, let's create a folder called test-project. Please, open up a terminal and change your directory into this folder.

$cd test-project/

Now let's turn this simple folder into a repository! To do that, we will use the command bellow to tell Git we want to initialise a repository in that folder.

$git init
1Initialized empty Git repository in /home/admin/test-project/.git/

Git has now created a .git directory in our repository. Let's create a few files, so Git has something to track!

$echo "lorem ipsum dolor sit amet" > file1.txt
$echo "ah lare diablo del paro" > file2.txt

Remember how I talked about the checkpoints we can create with Git? Now it is the time to create our first checkpoint. To do that, we will first need to tell Git which files to include. You can do that by either specifiying the name of the file we want to track like this...

$git add file1.txt

...or we can just tell Git to include all files in our repository in the checkpoint we will create soon:

$git add .

The files we will include here will be tracked, and the files that we don't include won't. This means that if we choose to track file1.txt, then create a checkpoint, then modify the data this file is storing, and then return back to the checkpoint, file1.txt will return to the state it was in when we made that checkpoint. If we choose not to include the file into the checkpoint, then modify the data inside of it, and then return back to the checkpoint, the data it this file will stay modified.

Anyways, Git now knows which files we want to include into this checkpoint. Now let's create the checkpoint itself! Before we do it though, I have been refering to commits as checkpoints, because it's easier to explain it this way. These 'checkpoints' are actually called commits, and we will create one just now using the command bellow.

$git commit -m "My first commit"

Let's break down this command. git commit is the command we are exectuing — the command that creates these checkpoints. Next it the -m flag. This is optional and means "message". By using this flag we tell Git that we will give this commit a name. After that there is the "My first commit" part, which is the name of this commit (checkpoint).

Cloning a repository from GitHub

So what does "cloning a repository" actually mean? It's simple! We will specify a repository we would like to clone and Git will make a copy of that repository from GitHub's servers into our local machine.

Let's clone a repository, shall we? This website it completly open source, this means that I released all the HTML and CSS code publicly for everyone to see on GitHub. Do you want a copy of it? No? Well, I will show you how to do it anyways!

$git clone https://github.com/nymvaclav/nymvaclav.github.io

git clone is the command that tells Git to clone a remote repository, and https://github.com/nymvaclav/nymvaclav.github.io is a URL address pointing to this repository on GitHub.

Final thoughts

As you can see, using Git is really straightforward and simple. In this article I showed you how to use some parts of Git's CLI, but there is much more to learn about Git. Also, if you do not like command line interfaces, GitHub released their version of Git with graphical user interface. I have tried using it already in the past, but I just prefer to use the CLI. I guess everyone has different preferances.
